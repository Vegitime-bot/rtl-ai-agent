# RTL AI Agent – Progress Log (2026-03-19)

## Summary

Neo4j graph data is now actively used during RTL generation (inference), not just ingested.

## Changes

### `scripts/neo4j_query.py` (new)

A reusable query helper that sits alongside `neo4j_ingest.py` and shares the same
`config/neo4j.yaml` + `NEO4J_PASSWORD` env-var connection pattern.

Key functions:

| Function | Description |
|---|---|
| `get_signal_drivers(module, signal)` | Signals with a `DRIVES` edge *into* the given signal |
| `get_signal_dependents(module, signal)` | Signals the given signal `DRIVES` |
| `get_causal_context(module, signals)` | 1-hop neighbors (drivers + dependents) for a list of signals |
| `format_graph_context(context)` | Formats the dict as a human-readable text block for LLM prompt injection |

CLI usage:
```bash
python scripts/neo4j_query.py --module tcon_basic --signal frame_done
```

### `orchestrator/flow.py` (augmented)

Before calling the LLM, the flow now:

1. Determines the primary module name from `causal_graph.json`.
2. Extracts Verilog-style identifiers from `findings` summaries and `plan` actions via a lightweight regex heuristic (`_extract_signals`), capped at 30 candidates.
3. Calls `get_causal_context` + `format_graph_context` from `neo4j_query`.
4. Prepends a `## Graph Context (from Neo4j)` block to the LLM prompt when context is available.
5. If Neo4j is unreachable, emits a `warnings.warn` and continues — the pipeline does **not** fail.

## Why

The causal graph already existed in Neo4j after the 2026-03-19 02:40 BUILD, but the LLM
was not aware of signal-level causality.  Feeding 1-hop graph context directly into the
prompt gives the model awareness of which signals gate, drive, or are driven by the signals
it is about to modify — reducing the risk of silent functional regressions.
